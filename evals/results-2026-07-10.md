# Eval run — 2026-07-10 (S7: structured-response-contract)

First run of S7, testing the newly added Structured Response Contract + templates +
phased-delivery + DevSecOps references (commit 4d547f6). N=1 per arm. **Directional,
not statistical.**

## Method
- **Treatment**: sonnet agent instructed to Read the installed SKILL.md from disk and
  follow it, routing to references itself. It was NOT told any grading criteria. It chose
  to load 5 relevant references (phased-delivery, devsecops, dotnet, architecture,
  review-templates) and skipped the other 8 — routing behaved as designed.
- **Control**: sonnet agent as "senior backend consultant", same task, no skill content.
- **Judge**: a separate opus agent, blind — neutral filenames (brief-K/brief-M, K=treatment,
  M=control), no information about how either was produced, scored strictly against the
  S7 rubric with quoted evidence per cell.

## Results (0–2 per metric)

| metric | control (M) | treatment (K) | delta |
|---|---|---|---|
| decision-structure | 1 | 2 | +1 |
| rejected-alternatives | 1 | 2 | +1 |
| phased-landing | 1 | 2 | +1 |
| grounded-honesty | 1 | 2 | +1 |
| security-governance-depth | 1 | 2 | +1 |
| **total** | **5** | **10** | **+5** |

Judge's adoption verdict: use K as the backbone ("decision-ready: every open question has
an owner, every phase has a gate and a rollback, the exec one-pager is ready to hand over"),
mine M for specific techniques.

## Findings
- **Sonnet discriminated this time.** The 2026-07-09 runs showed a ceiling on sonnet for
  judgment-trap scenarios (S1/S2/S5); S7 shows the contract layer adds measurable value on
  sonnet itself — the delta comes from STRUCTURE and GOVERNANCE (rejected-alternatives as a
  first-class section, per-phase exit criteria, role-named deciders, suppression/baseline
  governance, a filled exec one-pager), not from raw technical knowledge.
- **Control was technically strong** — in places more specific than treatment (explicit
  GitHub OIDC↔Azure AD workload identity federation, CDC/Debezium, canary percentages).
  The skill's contribution was turning comparable technical competence into a governable,
  trackable, decision-ready document. This matches the 2026-07-09 takeaway: the skill's
  value lives where no hard signal forces correctness — here, document completeness and
  decision governance.
- **Honesty rules visibly fired in treatment**: assumptions declared up front, deciders
  named as roles (never invented people), cost figures marked "draft — pending Phase 0"
  instead of fabricated.
- **One coverage miss in treatment**: the break-glass path (present in the pipeline
  template) did not surface in the answer's pipeline section.

## Caveats
- N=1 per arm; one scenario; re-run with N≥5 before quoting the delta.
- A pilot judge run leaked arm identity through filenames (skill-test-control.md) and was
  discarded for the headline result; directionally it agreed (control 7, treatment 10 —
  its stricter/looser control scoring vs the clean judge's 5 also shows judge variance of
  ±2 on the control arm, another reason not to overquote the exact delta).
- Both arms are consulting-style prompts with no repo; S7 tests document/decision quality,
  not repo execution.
- Judge blindness is label-level: a judge can still infer arm identity from style. Full
  blindness would need many mixed answers.

# S7 N=5 run with haiku arm — added 2026-07-10 (same day)

20 transcripts: {sonnet, haiku} × {control, treatment} × N=5 (sonnet run 1 reuses the
morning pair). Each transcript scored by an INDEPENDENT blind opus judge — neutral
filenames (ans-XX), single-transcript absolute scoring against the rubric, no information
about arms; all transcripts grep-checked for arm-identity leaks before judging (zero).
Break-glass promotion (4ffdee6) was live for the 4 new sonnet-T runs and all haiku-T runs.

## Results (cell mean over N=5, 0–2 per metric)

| cell | decision-structure | rejected-alts | phased-landing | grounded-honesty | security-governance | total |
|---|---|---|---|---|---|---|
| sonnet control | 2.0 | 1.6 | 2.0 | 1.0 | 1.0 | 7.6 |
| sonnet treatment | 2.0 | 2.0 | 2.0 | 2.0 | 2.0 | **10.0 (σ=0)** |
| haiku control | 2.0 | 1.0 | 1.2 | 1.0 | 1.0 | 6.2 |
| haiku treatment | 2.0 | 1.8 | 2.0 | 1.4 | 1.6 | 8.8 |

**Delta: sonnet +2.4, haiku +2.6.** Per-transcript table archived with the raw answers.

## Findings
- **The N=1 result replicates at N=5, on both models.** Treatment > control in all 10
  paired comparisons; no overlap between treatment and control score ranges within either
  model (sonnet 10 vs 7-8; haiku 8-9 vs 5-7).
- **Sonnet treatment is perfectly consistent**: 5/5 runs scored 10/10 under five different
  independent judges. The contract does not just raise the mean — it removes variance,
  which is what "rules a smaller model can execute reliably" was supposed to buy.
- **Where the value concentrates** (control→treatment): grounded-honesty 1.0→2.0 (sonnet)
  — assumptions declared, deciders as roles, numbers marked as estimates; security-
  governance 1.0→2.0 (sonnet) — suppression/baseline governance and rotation overlap
  windows appear only with the skill. For haiku, rejected-alternatives 1.0→1.8 and
  phased-landing 1.2→2.0 (control haiku plans lack systematic exit criteria).
- **decision-structure ceilinged in all four cells** (2.0 everywhere) — both models
  write well-sectioned answers unaided; this metric no longer discriminates and should be
  hardened (e.g., grade implementability of the design section, not section presence).
- **Break-glass regression check**: after promotion to Core (4ffdee6), judge evidence for
  ans-07 explicitly cites the emergency-path governance ("緊急放行留痕") — the 07-10
  morning miss did not recur in that run, though not every treatment answer surfaces it.
- **Honest cost finding: the contract makes answers LONGER.** All 5 haiku-treatment
  answers were judged too long for a "one-pager for my boss next week" ask (232-589
  lines); 3/5 sonnet-treatment answers too. The ten-section contract specifies sections,
  not volume, and weaker models compensate with bulk. Candidate fix for a future change:
  add a length-discipline line to the contract ("sections bound the SHAPE, not the size;
  match total volume to the ask").
- **Judge variance, quantified**: the same control transcript (sonnet run 1) scored 5
  (07-10 morning pairwise judge) and 7 (this run's absolute judge); treatment transcripts
  scored 10 under three independent judges across both protocols. Control scores carry
  ±1-2 judge noise; the treatment-control gap exceeds it.

## Caveats
- Same scenario S7 for all 20 runs — this measures the contract on ONE task shape;
  breadth still needs the repo-backed variant (next step 2).
- Absolute single-transcript judging (this run) vs pairwise judging (morning run) give
  slightly different control scores; deltas are computed within-protocol only.
- Judges are label-blind, not style-blind.

## Remaining next steps
1. ~~Repo-backed S7 variant~~ — done same day as S8, see below.
2. Harden decision-structure metric (ceilinged everywhere).
3. Consider a length-discipline line in the Structured Response Contract (see cost
   finding) — weigh against prompt-bloat before adding.

## Findings
- **With tool access, both arms ground well unaided**: every transcript scored 2 on
  citation grounding (zero fabricated paths across 8 runs, judge-verified), and nearly
  everyone found the FK blocker and both secret locations. This confirms the 2026-07-09
  consolidated insight from the other direction: traps visible in readable code are
  hard-signal traps, and hard-signal traps ceiling regardless of the skill.
- **The one soft-signal trap (t4 check-then-insert race) was missed by almost everyone**
  — 6/8 scored 0-1 with no specific detection; only sonnet-treatment r1 partially
  surfaced it. Notable: the skill's own gate matrix lists check-then-write races, and the
  treatment agents had it on disk — but the review-prevention content did not activate
  during a *design-consulting* task. Candidate skill improvement: the phased-delivery
  Phase 0 inventory step could instruct running the pre-landing gate lenses over the code
  being inventoried, so review-grade findings surface during planning, not just review.
- **haiku-treatment r2 was the worst transcript (8/12)**: it spent its budget following
  the contract's structure and missed the appsettings secrets entirely (found only the CI
  yaml one). For weak models on repo tasks, contract adherence can compete with code
  inspection — consistent with the S7 verbosity finding.
- Combined with S7: the skill's measurable value is in decision/document/governance
  quality (S7: +2.4/+2.6), NOT in code-reading thoroughness (S8: ≈0). That is what its
  design intends ("decision pressure", review gates, templates) — but S8 draws the
  boundary honestly.

## Caveats
- N=2 per cell — directional only; t4 uniform-miss is the most robust observation
  (8/8 failed to fully name the planted race).
- The consulting framing does not explicitly ask for a code review; t4 partly measures
  whether review lenses self-activate, which is a stretch goal for this scenario.
- Both arms saw the fixture path inside the skill's own repo; two haiku-treatment
  transcripts used the outer repo name in OIDC subject-claim examples. Not an
  arm-identity leak (both arms saw the same path), recorded for completeness.

## The headline finding: task framing decides which lenses activate — in BOTH arms

- Design mode (S8): FK split-blocker found 8/8, race fully named 0/8.
- Review mode (S9): race fully named 6/8 — **and the FK blocker flagged 0/8**, including
  every treatment run. Reviewing "for landmines before go-live" surfaces code defects and
  hides architectural blockers; consulting "how do we split this" does the opposite. The
  blindness is symmetric and arm-independent: models answer the question they were asked,
  and the skill (pre-fix) did not bridge modes.
- The Phase 0 gate-lens change bridges one direction (review lenses now fire during
  design work: S8v2). The other direction — architectural-blocker lenses during review —
  is a candidate future change (e.g., a Review Stance line: "if the repo is under an
  announced initiative, findings that block that initiative are in scope"), left
  unimplemented pending evidence it matters in practice.
