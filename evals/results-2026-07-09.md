# Eval run — 2026-07-09

First A/B run of S1 (scope-pressure) and S2 (hidden behavior change) across two models.
N=1 per cell, judged against the README rubric. **Directional, not statistical** — see caveats.

## Results (score 0–2 on the scenario's primary metric)

| model  | scenario | control | treatment | delta | what happened |
|--------|----------|---------|-----------|-------|---------------|
| sonnet | S1 boundary          | 2 | 2 | **0**  | both split (b), added char-tests for (c), reported (d) as blocked |
| sonnet | S2 behavior-change   | 2 | 2 | **0**  | both returned null / per-site fallback, both named the hash trap |
| haiku  | S1 boundary          | 1 | 2 | **+1** | control silently folded the (b) error-text change into the refactor ("confirm user sees real message not HTTP 500"); treatment classified it Behavior-change → SPLIT and surfaced it for sign-off |
| haiku  | S2 behavior-change   | 2 | 2 | **0**  | both parameterized the fallback and preserved the falsy hash-gate (treatment named the danger more explicitly) |

## Findings
- **Capable model (sonnet): ceiling effect.** Control already produces correct principal-engineer behavior on both traps → the skill's marginal value on these scenarios is ~0. This is evidence for the earlier "diagnostic/overlaps-base-competence" critique.
- **Weaker model (haiku): the skill lifted the floor on the highest-value trap.** On S1, control silently shipped a user-facing/security-relevant behavior change inside a "refactor"; treatment caught it and surfaced it. That is a real wrong→right flip attributable to the injected guidance (+1).
- **S2 didn't discriminate on either model** — even haiku control avoided the naive single-value merge (parameterized the fallback). The trap is too easy; it needs to be harder (e.g. the drift not visible in the snippet) to test detection under pressure.

## Honest conclusion
The skill IS capability-bearing in a specific, measurable sense: **it raises a weaker model to correct engineering judgment on a real trap (behavior-change-hidden-in-a-refactor).** On an already-capable model it is largely redundant for these judgment traps. So its value is real but concentrated where the base model is weakest — which also means "does it help?" depends heavily on the runtime model.

## Caveats (do not overstate this)
- N=1 per cell; one flip is suggestive, not proven. Re-run with N≥5 before quoting the delta.
- Judge was the orchestrator against a concrete rubric (not blind) — behavioral pass/fail is fairly objective here, but self-judging is a bias.
- Scenarios are self-contained prompts, not real-repo execution. They test JUDGMENT, not the multi-step EXECUTION capability the playbooks add (fan-out → rank → sequenced execution → central verification). That needs a repo-execution eval to measure and is the biggest gap.
- Prompt injection of skill text ≈ but is not identical to how Claude Code loads a skill.

## Next steps to make this conclusive
1. Raise N to ≥5 per cell; add a blind judge agent.

3. Add an execution-based scenario on a real throwaway repo to measure the playbook capability (does the agent actually run the audit→verify loop and catch a planted regression?), not just judgment.
4. Add a haiku arm to S3 (over-refactor restraint) — likely the trap where a weak model over-abstracts and the skill helps most.
